\begin{frame}
  \frametitle{On the Impact of \alert{Formal Verification} on \alert{Software Development}}
  \begin{changemargin}{1cm}
Auto-active verifiers like Dafny aim to make formal methods accessible to non-expert users through SMT automation. However, despite the automation and other programmer-friendly features, they remain \alert{sparsely used in real-world software development}, due to the significant effort required to apply them in practice. We interviewed \alert{14 experienced Dafny users} about their experiences using it in \alert{large-scale projects}. We apply grounded theory to analyze the interviews to \alert{systematically identify how auto-active verification impacts software development}, and to identify opportunities to simplify the use, and hence, expand the adoption of verification in software development.
  \end{changemargin}
\end{frame}

\begin{frame}
  \frametitle{Summary of Results}
    \large
    \begin{changemargin}{1cm}
      \begin{itemize}
    \item two types of developers: \emph{formal-first} vs \emph{engineering-first}\\[1em]
    \item design/implementation: verification adds specification writing to design phase, and specification-proving to implementation phase. \\[1em]
      also in implementation: debugging failed proof attempts, hardening the code for more robust verification.\\[1em]
    \item testing/deployment: verification affects testing \& code reviews, making reviewing harder.
      \end{itemize}
    \end{changemargin}

\end{frame}

\begin{frame}
  \frametitle{Specification styles}
  \Large
  \begin{changemargin}{1cm}
    \texttt{method} or \texttt{function}?\\[2em]
    Functional specs in 5 of the interviews:\\
    no need to track what state has changed; aliasing; complex control-flow.\\[2em]
    Previous lecture: \texttt{swap}, definitely imperative.\\
    Does in-place modification of state; nothing else changes.\\[1em]
    Functional \texttt{swap}: return a copy of the array.
  \end{changemargin}
\end{frame}

\begin{frame}[fragile]
  \frametitle{Figure 3 example }

  \begin{lstlisting}[language=dafny]
method UpperBoundedAdditionImpl(x:uint64, y:uint64, u:uint64)
       returns (sum:uint64)
  ensures sum as int = UpperBoundedAddition(x as int, y as int, 
       UpperBoundFinite(u as int));
{
  if y >= u {
    sum := u; // write
  } else if x >= u - y {
    sum := u; // write 
  } else {
    sum := x + y; // write
  }
}

// functional spec: no intermediate state
function UpperBoundedAddition(x:int, y:int, u:UpperBound):int
{
  // LeqUpperBound checks if a given integer x
  // is less than or equal to the specified upper bound u.
  if LeqUpperBound(x + y, u) then x + y else u.n
}
\end{lstlisting}

\end{frame}

\begin{frame}[fragile]
  \frametitle{Problem: aliasing}
  \Large
\begin{lstlisting}[language=dafny]
/** A helper method to swap two positions in an array */
method swap(a: array<int>, pos1:nat, pos2:nat)
\end{lstlisting}

\begin{changemargin}{1cm}
  OK, array \texttt{a} changes, but are there any aliases of it that simultaenously change?\\[1em]
  Rust: uniqueness, enforced.\\
  C++: uniqueness is an option.
  \end{changemargin}
\end{frame}

\begin{frame}

  \begin{changemargin}{1cm}

\begin{frame}
  \frametitle{Complaints \& Workarounds}
  \Large
  \begin{changemargin}{1cm}
    \begin{itemize}
    \item Dafny error messages don't say why;
    \item used assertions to figure out what Dafny knows (like breakpoints);
    \item used \texttt{assume} and conditionals to understand what Dafny knows.
    \end{itemize}
    ~\\[1em]
    Wider challenge: ``build a mental model of implicit verifier state''.\\
    Interactive verifiers easier to use in that sense; \\
    for Dafny, annotations can help.
  \end{changemargin}
\end{frame}

\begin{frame}
  \frametitle{Proof Hardening}
  \Large
  \begin{changemargin}{1cm}
    Kind of like code refactoring to eliminate smells.\\[2em]
    Proof works now, but might not work on another machine, another solver version, or after small impl change.\\[1em]
    \begin{quote}
some little fiddly change to the code,
just maybe pass one more element in the state [\ldots] and all the proofs
would just stop working.
    \end{quote}

    ``soul crushing''; induces ``existential dread''
  \end{changemargin}
\end{frame}

\begin{frame}
  \frametitle{Brittleness}
  \Large
  \begin{changemargin}{1cm}
    3 participants had not encountered brittle proofs.\\[1em]
    Some addressed brittleness reactively: increase resource limits.\\
    Others proactively: month-long major refactoring; or, fix timeouts before continuing.
  \end{changemargin}
\end{frame}

\begin{frame}
  \frametitle{Why brittleness?}
  \Large
  \begin{changemargin}{1cm}
    Dafny doesn't know what's important.\\
    Passes too much context to server, but sometimes the wrong context.\\[1em]
    Try to make state visible, but gigabytes of state is unreadable.\\[1em]
    Frame conditions also get large.
  \end{changemargin}
\end{frame}

\begin{frame}
  \frametitle{Dealing with brittleness}
  \Large
  \begin{changemargin}{1cm}
    Many: monitor solver resource count to see if things getting hard.\\[1em]
    Also: add or remove hint assertions, reducing solving time.\\[1em]
    Other tools:
    \begin{itemize}
    \item ``unsat core''
    \item \texttt{assert false}
    \item using smaller lemmas;
    \item \texttt{opaque} and \texttt{reveal}
    \end{itemize}
  \end{changemargin}
\end{frame}

\begin{frame}
  \frametitle{Dealing with brittleness II}
  \Large
  \begin{changemargin}{1cm}
    More systematically: 4 used structured approaches, including style guides, to reduce automation.\\[1em]
    Report: Dafny is getting better.\\[2em]
    Brittleness only matters to you if you observe it.
  \end{changemargin}
\end{frame}

\begin{frame}
  \frametitle{How Testing}
  \Large
  \begin{changemargin}{1cm}
    Everyone does it.\\
    Half compare spec to impl or to expected behaviour.\\
    \begin{itemize}
    \item \emph{verified specifications}: some just wanted executable model of the code; not actually deployed, but model compared to impl.
    \item \emph{verified implementations}: verified code that would be deployed, use testing to check vs unverified legacy code.
    \end{itemize}
    Differential random testing applicable.\\[1em]
    Additionally: some verified new implementations of new specifications; there was no legacy.
  \end{changemargin}
\end{frame}

\begin{frame}
  \frametitle{Unit versus end-to-end testing}
  \Large
  \begin{changemargin}{1cm}
    e.g. JUnit used all the time for system tests.\\[1em]
    Participants did both unit testing and end-to-end testing, though
    what infrastructure is unclear.\\[1em]
  \end{changemargin}
\end{frame}

\begin{frame}
  \frametitle{Real Data}
  \Large
  \begin{changemargin}{1cm}
    One participant ran verified impl in parallel with prod, finding differences, even after fuzzing.\\[1em]
    Another participant said people wanted to see verified code working on real data.\\[2em]
    NB: Dafny can be transpiled to other languages \& tested for performance.
  \end{changemargin}
\end{frame}

\begin{frame}
  \frametitle{Code Review}
  \Large
  \begin{changemargin}{1cm}
    Is a best practice; all multi-author projects here used it.\\[2em]
    But: makes reviews larger. Need to review specs \& transpiled code, plus impls.\\
    Can be ``quite large'', and proof changes intertwined with code.\\[1em]
    Participants focussed on specifications, plus non-verified code (e.g. libraries).\\[1em]
  \end{changemargin}
\end{frame}

\begin{frame}
  \frametitle{What to Review}
  \Large
  \begin{changemargin}{1cm}
    One participant reviewed proofs for clarity \& maintainability.\\[1em]
    Sometimes participants reviewed transpiled code (due to lack of trust), but not all the time.\\[1em]
    Especially likely when other teams would interact with transpiled code.
  \end{changemargin}
\end{frame}

\begin{frame}
  \frametitle{Impact of Verification on Maintenance}
  \Large
  \begin{changemargin}{1cm}
    Testing: confidently refactor \& optimize code.\\
    (Rust: ``fearless concurrency'' due to type system).\\[2em]
    Verification: even better; can rely on verified specification even more than on your test suite; Dafny will tell you of divergence.\\[2em]
    A change will often require both spec \& proof changes.\\
    But, get more knowledge of the system as a result: what might break?
  \end{changemargin}
\end{frame}

\begin{frame}
  \frametitle{Breaking changes}
  \Large
  \begin{changemargin}{1cm}
    Can see which spec no longer holds, and focus testing accordingly.\\[1em]
    Use small fixes for spec and proof.\\[2em]
    And, get some confidence that a change is not breaking if Dafny agrees.\\[2em]
    All this relies on verified code being deployed prior to change.
  \end{changemargin}
\end{frame}
    
\begin{frame}
  \frametitle{Optimization}
  \Large
  \begin{changemargin}{1cm}
    Participants reported being more willing to make changes due to verification.\\[1em]
    e.g. localized changes that do not affect the specification, such as algorithmic optimization and removing redundant checks.\\[2em]
    Verification implies that new code continues to meet the specification.\\[1em]
    (Redundant checks: those that can be shown at proof time, eg with \texttt{requires}).
  \end{changemargin}
\end{frame}

\begin{frame}
  \frametitle{Summary}
  \Large
  \begin{changemargin}{1cm}
    Yes, people use Dafny on real systems, perhaps via transpilers.\\[2em]
    Affects all stages of software developments.\\
    Results in software that meets its formally-expressed specifications, and enables lower-risk changes.
  \end{changemargin}
\end{frame}
